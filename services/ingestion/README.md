# Ingestion Service Layout

This service is the future backend worker that builds the park knowledge database.

The mobile app should never scrape sources directly. It should consume a versioned SQLite bundle and backend APIs generated by this service.

Generated raw files and manifests should land under `data/ingestion/`, which is ignored by git.

## Ownership

This service owns:

- fetching raw park source data
- normalizing source data into canonical park records
- validating catalog quality
- exporting the offline SQLite bundle
- emitting curated documents for Bedrock Knowledge Bases

This service does not own:

- chat orchestration in the mobile app
- user trail history
- live device location

## Run Order

1. `fetch-park-listing`
   - establishes the expected unit list

2. `fetch-open-data`
   - downloads official spatial and metadata datasets

3. `fetch-park-pages`
   - snapshots each park detail page and linked documents

4. `parse-park-page`
   - extracts deterministic, structured fields

5. `parse-document`
   - handles brochures or PDF content
   - may call Bedrock for irregular layouts

6. `build-canonical-park`
   - joins all normalized inputs into one park contract

7. `validate-catalog`
   - blocks export if quality gates fail

8. `build-sqlite-bundle`
   - produces the mobile offline database

9. `publish-offline-bundle`
   - publishes the current SQLite bundle to stable app-facing S3 keys and manifest files

10. `sync-to-s3`
   - mirrors local ingestion outputs into S3 for durable storage

11. `build-rag-corpus`
   - produces citation-friendly documents for Bedrock

## Output Contracts

The ingestion service should write four data products:

1. Raw source snapshots
   - immutable
   - versioned by fetch time and checksum

2. Canonical JSON records
   - one structured file per park
   - shape defined in [types/parks.ts](../../types/parks.ts)

3. SQLite bundle
   - schema defined in [docs/sqlite-schema.sql](../../docs/sqlite-schema.sql)

4. RAG corpus
   - compact markdown or JSONL documents
   - one document per park topic or safety topic

## Implementation Notes

- Prefer GeoJSON and CSV downloads over shapefiles to avoid unnecessary GIS tooling early on.
- Keep raw HTML and PDFs so parser bugs can be reproduced locally.
- Fail closed on stale operational data instead of guessing.
- Keep the canonical park ID stable across DynamoDB, SQLite, and RAG documents.
- The first runnable jobs are `npm run ingest:park-listing`, `npm run ingest:open-data`, and `npm run ingest:park-pages`.
- The next local transform job is `npm run parse:park-pages`.
- The first canonical build job is `npm run build:canonical-parks`.
- The first QA validation job is `npm run qa:validate-catalog`.
- The first offline export job is `npm run export:sqlite-bundle`.
- The first app-delivery publication job is `npm run publish:offline-bundle`.
- The first S3 publishing job is `npm run sync:ingestion-to-s3`.
